# 出発見守り和子さん｜最終仕様書（完全版）

## 0. 概要

### システム名
出発見守り和子さん

### 目的
出勤前の確認を自動化し、問題が起きそうな人だけを管制が対応する

### 手段
- LINE（ボタン）＋ 自動電話 ＋ Googleスプレッドシート

### 基本方針
- 判定は出発報告ボタンを押した時刻のみ
- 電話の着信・応答・留守電は判定に使わない
- 管制は最終段階のみ介入

---

## 1. 役割

### 1.1 キャスト（現場）
- **前日**：出発予定時間を登録
- **当日**：出発したら出発報告ボタンを押す
- **それ以外**：何もしない

### 1.2 和子さん（システム）
- 未登録／未報告を自動判定
- 必要な人にだけ通知・電話
- 状態をスプレッドシートに記録

### 1.3 管制
- 自動処理で解決しないケースのみ対応

---

## 2. 用語定義

| 用語 | 定義 | データ型 | 形式・制約 |
|------|------|----------|------------|
| **出発予定時間** | 前日に登録する時間 | datetime | JST（Asia/Tokyo）、時:分（5分単位）、例: "08:30" |
| **出発時間** | LINEの出発報告ボタンを押した時刻 | datetime | JST（Asia/Tokyo）、ISO 8601形式、ミリ秒まで記録 |
| **出発判定** | 出発時間と出発予定時間の比較結果 | enum | "OK" / "遅れ返" / "要確認" |
| **通常出発予定時間** | キャスト一覧に登録されているデフォルト時間 | time | HH:MM形式、5分単位 |

**重要**：判定は出発報告ボタン押下時刻のみを使用

## 3. 前日運用（厳密な時系列）

### 3.1 出発予定時間の登録

#### 3.1.1 登録方法
- LINEで時 → 分の2タップ
- 5分単位のみ
- 文字入力なし

#### 3.1.2 データ形式
- **時**: 0-23の整数
- **分**: 0, 5, 10, 15, 20, 25, 30, 35, 40, 45, 50, 55（5分単位のみ）
- **日付**: 登録日の翌日（システムが自動判定）

#### 3.1.3 バリデーション
- 時が0-23の範囲外 → エラーメッセージ表示
- 分が5分単位でない → エラーメッセージ表示
- 登録済みの場合 → 上書き確認（または自動上書き）

#### 3.1.4 保存先
- Googleスプレッドシート「出発見守り_当日管理」シート
- 列: `日付`, `氏名`, `LINE_ID`, `出発予定時間`

### 3.2 リマインド／管制通知

| 時刻 | システム動作 | 対象 | 通知内容 | エラー時の処理 |
|------|------------|------|----------|---------------|
| 20:00 | 未登録者にLINE通知 | 出発予定時間未登録の全キャスト | 「明日の出発予定時間を登録してください」 | ログ記録、リトライなし |
| 21:00 | 未登録者にLINE通知 | 同上 | 同上 | ログ記録、リトライなし |
| 22:00 | 未登録者にLINE通知 | 同上 | 同上 | ログ記録、リトライなし |
| 22:30 | 管制へ未登録者通知 | 管制担当者 | 未登録者人数＋氏名リスト | ログ記録、リトライなし |
| 23:00 | 未登録者に最終LINE通知 | 出発予定時間未登録の全キャスト | 「明日の出発予定時間を登録してください（最終）」 | ログ記録、リトライなし |
| 24:00 | 通常出発予定時間を自動採用／無ければ管制通知 | システム | 未登録者に通常出発予定時間を設定、通常出発予定時間も無い場合は管制通知 | ログ記録、管制通知失敗時はアラート |

#### 3.2.1 未登録者の判定条件
- `出発見守り_当日管理`シートで、翌日の日付かつ該当キャストの`出発予定時間`が空欄またはNULL

#### 3.2.2 通常出発予定時間の自動採用
- `キャスト一覧`シートの`通常出発予定時間`を`出発見守り_当日管理`シートにコピー
- `通常出発予定時間`も空欄の場合は管制通知

#### 3.2.3 タイムゾーン
- すべてJST（Asia/Tokyo）で処理
- 日付境界（23:59:59 → 00:00:00）は翌日として扱う

## 4. 当日運用（出発）

### 4.1 キャスト操作（確定文言）

**操作**: 出発したら、LINEで「出発報告」ボタンを押す

#### 4.1.1 ボタン仕様
- **ボタン名**: "出発報告"
- **ボタンタイプ**: Postback（サーバーにデータ送信）
- **重複押下**: 2回目以降は「既に出発報告済みです」と表示、処理はスキップ

#### 4.1.2 データ送信
- Postback data: `{"action": "departure_report", "timestamp": "YYYY-MM-DDTHH:mm:ss.SSS+09:00"}`

### 4.2 判定ロジック（完全版）

#### 4.2.1 判定フロー

```
IF 出発報告ボタンが押された:
  → 出発時間 = ボタン押下時刻（JST）
  → 出発予定時間 = スプレッドシートから取得（JST）
  
  IF 出発時間 <= 出発予定時間:
    → 判定 = "OK"
    → 次の動き = 何もしない
    → スプレッドシート更新: 出発判定 = "OK"
    
  ELSE IF 出発時間 > 出発予定時間:
    → 判定 = "遅れ返"
    → 次の動き = 原則何もしない（ログのみ）
    → スプレッドシート更新: 出発判定 = "遅れ返"
    
  ELSE IF 出発予定時間までに出発報告が無い:
    → 判定 = "要確認"
    → 次の動き = 自動電話へ
    → スプレッドシート更新: 出発判定 = "要確認"
```

#### 4.2.2 判定テーブル

| 状態 | 判定 | 次の動き | スプレッドシート更新 |
|------|------|----------|---------------------|
| 出発予定時間までに報告あり | OK | 何もしない | `出発判定` = "OK", `出発時間` = 報告時刻 |
| 出発予定時間以降に報告 | 遅れ返 | 原則何もしない | `出発判定` = "遅れ返", `出発時間` = 報告時刻 |
| 出発予定時間までに未報告 | 要確認 | 自動電話へ | `出発判定` = "要確認"（出発時間は空欄） |

**重要**: 遅れ返でも自動電話はしない

#### 4.2.3 時刻比較の詳細
- 比較は日時（datetime）で行う
- 例: 出発予定時間が "08:30" の場合、当日の "2024-01-15 08:30:00" と比較
- ミリ秒単位まで記録するが、比較は秒単位で行う（ミリ秒は切り捨て）

## 5. 自動電話（出発見守り）

### 5.1 トリガー
- 出発予定時間までに出発報告が無い

### 5.2 電話フロー（確定）

#### 5.2.1 電話①
- **間隔**: 5分おき
- **回数**: 5回
- **合計時間**: 約25分（最初の電話から最後の電話まで）

#### 5.2.2 電話②
- **間隔**: 3分おき
- **回数**: 10回
- **合計時間**: 約30分（最初の電話から最後の電話まで）

#### 5.2.3 全体
- **合計**: 約45分で終了
- **電話番号**: `キャスト一覧`シートの`電話番号`列から取得

### 5.3 電話の詳細仕様

#### 5.3.1 音声メッセージ内容
```
「おはようございます。出発見守り和子さんです。本日の出発報告をお願いします。LINEの出発報告ボタンを押してください。」
```

#### 5.3.2 電話番号形式
- **形式**: 国際形式（E.164）
- **例**: +819012345678（日本の場合、+81 + 0を除いた番号）
- **バリデーション**: 正規表現 `^\+[1-9]\d{1,14}$`

#### 5.3.3 電話の結果処理
- **着信**: ログに記録（判定には使わない）
- **応答**: ログに記録（判定には使わない）
- **留守電**: ログに記録（判定には使わない）
- **不通**: ログに記録（判定には使わない）
- **重要**: 電話の結果は判定に一切使わない。出発報告ボタンのみが判定材料

#### 5.3.4 電話実行タイミング
- 出発予定時間の1分後から開始（例: 08:30が予定なら08:31から）
- 電話①: 08:31, 08:36, 08:41, 08:46, 08:51
- 電話②: 08:54, 08:57, 09:00, 09:03, 09:06, 09:09, 09:12, 09:15, 09:18, 09:21

#### 5.3.5 電話中に出発報告があった場合
- 残りの電話は即座にキャンセル
- スプレッドシート更新: `出発判定` = "OK" または "遅れ返"（時刻で判定）

### 5.4 位置づけ
- 出発報告を促すための手段
- 応答・留守電は判定に使わない

## 6. 管制が動く条件（厳密）

### 6.1 条件（すべて該当する場合のみ）

1. 出発予定時間までに出発報告なし
2. 電話①②を完走
3. それでも出発報告が無い

### 6.2 管制対応

#### 6.2.1 対応内容
- 人の手で電話
- 状況確認（遅刻／体調不良／事故等）
- 穴埋め・現場調整

#### 6.2.2 通知タイミング
- 電話②の最後の電話が終了した時点（出発報告がまだ無い場合）
- 即座にLINE通知（緊急アラート）

#### 6.2.3 通知内容
```
【緊急アラート】
氏名: [氏名]
出発予定時間: [HH:MM]
現在時刻: [YYYY-MM-DD HH:MM]
状況: 電話①②完走後も出発報告なし
```

## 8. データ管理（Googleスプレッドシート）

### 8.1 シート①：キャスト一覧

#### 8.1.1 列定義

| 列名 | データ型 | 必須 | 形式・制約 | 説明 |
|------|----------|------|------------|------|
| 氏名 | string | Yes | 最大100文字 | キャストの氏名 |
| LINE_ID | string | Yes | 最大100文字、英数字とアンダースコアのみ | LINE Messaging APIのユーザーID |
| 電話番号 | string | Yes | E.164形式（+819012345678） | Twilioで使用する電話番号 |
| 通常出発予定時間 | time | No | HH:MM形式、5分単位 | デフォルトの出発予定時間 |
| 所属 | string | No | 最大100文字 | キャストの所属 |
| 備考 | string | No | 最大500文字 | その他の備考 |

#### 8.1.2 データ整合性
- `LINE_ID`は一意であること（重複不可）
- `電話番号`は一意であること（重複不可）
- 行の削除は論理削除（削除フラグ列を追加するか、別シートに移動）

### 8.2 シート②：出発見守り_当日管理

#### 8.2.1 列定義

| 列名 | データ型 | 必須 | 形式・制約 | 説明 |
|------|----------|------|------------|------|
| 日付 | date | Yes | YYYY-MM-DD形式 | 対象日 |
| 氏名 | string | Yes | 最大100文字 | キャストの氏名（キャスト一覧と一致） |
| LINE_ID | string | Yes | 最大100文字 | LINE Messaging APIのユーザーID |
| 出発予定時間 | datetime | No | YYYY-MM-DD HH:MM:SS形式（JST） | 前日に登録された出発予定時間 |
| 出発時間 | datetime | No | YYYY-MM-DD HH:MM:SS.SSS形式（JST） | 出発報告ボタン押下時刻 |
| 出発判定 | enum | No | "OK" / "遅れ返" / "要確認" / "管制対応" | 判定結果 |
| 出発電話回数 | integer | No | 0 / 1 / 2 | 0=電話なし、1=電話①のみ、2=電話①②完走 |
| 最終結果 | enum | No | "出勤OK" / "遅刻" / "穴埋め" / "未確定" | 最終的な結果（管制が記入） |
| 管制メモ | string | No | 最大1000文字 | 管制が記入するメモ |

#### 8.2.2 データ整合性
- `日付` + `LINE_ID`の組み合わせは一意であること（1日1人1レコード）
- `出発時間`が記録されている場合、`出発判定`は必ず "OK" または "遅れ返"
- `出発判定`が "要確認" の場合、`出発時間`は空欄
- `出発電話回数`が 2 の場合、`出発判定`は "管制対応" または "OK"/"遅れ返"（電話中に報告があった場合）

#### 8.2.3 データ更新タイミング
- **出発予定時間**: 前日の登録時、または24:00の自動採用時
- **出発時間**: 出発報告ボタン押下時
- **出発判定**: 出発報告ボタン押下時、または出発予定時間経過時
- **出発電話回数**: 電話①開始時=1、電話②開始時=2
- **最終結果**: 管制が手動で記入
- **管制メモ**: 管制が手動で記入

## 9. 判定ロジック（固定・完全版）

### 9.1 擬似コード

```python
def 判定ロジック(出発時間: datetime, 出発予定時間: datetime) -> str:
    """
    出発判定ロジック
    
    Args:
        出発時間: 出発報告ボタン押下時刻（JST）
        出発予定時間: 出発予定時間（JST）
    
    Returns:
        "OK" / "遅れ返" / "要確認"
    """
    if 出発時間 is None:
        # 出発報告が無い
        if 現在時刻 > 出発予定時間:
            return "要確認"
        else:
            return None  # まだ判定できない
    
    # 時刻比較（秒単位、ミリ秒は切り捨て）
    出発時間_秒 = 出発時間.replace(microsecond=0)
    出発予定時間_秒 = 出発予定時間.replace(microsecond=0)
    
    if 出発時間_秒 <= 出発予定時間_秒:
        return "OK"
    else:
        return "遅れ返"
```

### 9.2 状態遷移図

```
[未登録]
  ↓ (前日登録)
[出発予定時間登録済み]
  ↓ (当日、出発予定時間前)
[待機中]
  ↓
  ├─ (出発報告ボタン押下、出発予定時間前) → [OK] → 終了
  ├─ (出発報告ボタン押下、出発予定時間後) → [遅れ返] → 終了
  └─ (出発予定時間経過、未報告) → [要確認] → [自動電話①] → [自動電話②] → [管制対応]
```

### 9.3 エッジケース

#### 9.3.1 出発予定時間と出発時間が完全に一致
- 判定: "OK"（<= なので）

#### 9.3.2 出発報告が複数回押された
- 最初の1回のみ有効、2回目以降は「既に出発報告済みです」と表示

#### 9.3.3 出発予定時間が登録されていない
- 24:00に通常出発予定時間を自動採用
- 通常出発予定時間も無い場合は管制通知

#### 9.3.4 電話中に出発報告があった
- 残りの電話は即座にキャンセル
- 判定は出発時間と出発予定時間の比較で決定

## 12. エラーハンドリング・例外処理

### 12.1 LINE API エラー

#### 12.1.1 エラー種類と処理

| エラーコード | エラー内容 | 処理 | リトライ |
|------------|-----------|------|---------|
| 400 | リクエスト不正 | ログ記録、ユーザーにエラーメッセージ送信 | なし |
| 401 | 認証エラー | ログ記録、アラート通知 | なし |
| 403 | 権限エラー | ログ記録、アラート通知 | なし |
| 429 | レート制限 | ログ記録、指数バックオフでリトライ（最大3回） | あり |
| 500 | サーバーエラー | ログ記録、指数バックオフでリトライ（最大3回） | あり |
| タイムアウト | 接続タイムアウト | ログ記録、リトライ（最大3回） | あり |

#### 12.1.2 リトライロジック
- 指数バックオフ: 1秒、2秒、4秒
- 最大3回までリトライ
- 3回失敗した場合はログ記録のみ

### 12.2 Twilio API エラー

#### 12.2.1 エラー種類と処理

| エラーコード | エラー内容 | 処理 | リトライ |
|------------|-----------|------|---------|
| 20003 | 無効な電話番号 | ログ記録、スプレッドシートにエラーフラグ | なし |
| 21211 | 無効な電話番号形式 | ログ記録、スプレッドシートにエラーフラグ | なし |
| 21608 | 無効な電話番号 | ログ記録、スプレッドシートにエラーフラグ | なし |
| 20001 | 認証エラー | ログ記録、アラート通知 | なし |
| 20429 | レート制限 | ログ記録、指数バックオフでリトライ（最大3回） | あり |
| 500 | サーバーエラー | ログ記録、指数バックオフでリトライ（最大3回） | あり |
| タイムアウト | 接続タイムアウト | ログ記録、リトライ（最大3回） | あり |

#### 12.2.2 電話発信失敗時の処理
- ログに記録
- スプレッドシートの`管制メモ`に「電話発信失敗」と記録
- 次の電話は通常通り実行（スキップしない）

### 12.3 Google Sheets API エラー

#### 12.3.1 エラー種類と処理

| エラーコード | エラー内容 | 処理 | リトライ |
|------------|-----------|------|---------|
| 400 | リクエスト不正 | ログ記録、アラート通知 | なし |
| 401 | 認証エラー | ログ記録、アラート通知 | なし |
| 403 | 権限エラー | ログ記録、アラート通知 | なし |
| 429 | レート制限 | ログ記録、指数バックオフでリトライ（最大5回） | あり |
| 500 | サーバーエラー | ログ記録、指数バックオフでリトライ（最大5回） | あり |
| 503 | サービス利用不可 | ログ記録、指数バックオフでリトライ（最大5回） | あり |
| タイムアウト | 接続タイムアウト | ログ記録、リトライ（最大5回） | あり |

#### 12.3.2 スプレッドシート書き込み失敗時の処理
- ログに記録
- リトライ（最大5回）
- 5回失敗した場合は、メモリ上に保持して後で再試行（キューに保存）
- アラート通知

### 12.4 ネットワークエラー

#### 12.4.1 処理
- すべてのAPI呼び出しでタイムアウトを設定（30秒）
- タイムアウト時はリトライ（各APIのリトライロジックに従う）
- 3回連続でタイムアウトした場合は、ログ記録のみ

### 12.5 データ整合性エラー

#### 12.5.1 処理
- スプレッドシートから取得したデータのバリデーション
- 不正なデータ（NULL、形式不正など）はログ記録し、デフォルト値を使用
- 例: `出発予定時間`がNULLの場合、`通常出発予定時間`を使用
